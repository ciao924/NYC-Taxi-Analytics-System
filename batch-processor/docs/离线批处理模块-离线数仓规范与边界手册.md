## 修正后的离线数仓规范与边界手册（含数据检测模块）

本手册基于原版修订，修正了命名笔误、缓存描述不准确、质量阈值与代码实现不一致等问题，并新增 **§ 3\.7 数据质量检测与监控规范**，适配 MySQL 表结构 `nyc\_taxi\_quality`。

---

### 3\.1 分层边界规范

|层级|职责|约束|输入|输出|
|---|---|---|---|---|
|**ODS**|原始数据入湖、分区管理、格式统一|不做任何过滤、清洗、转换|Parquet/CSV 源文件|Hive/Iceberg 原始表，按 `year, month` 分区|
|**DWD**|数据清洗（空值、异常值）、维度关联、生成唯一标识|保留全量明细，不做聚合|ODS 原始表 \+ 维度表|事实宽表，按 `year, month, taxi\_type` 分区|
|**DWS**|轻度汇总（日 / 小时级）、多维度预聚合|粒度固定，不跨业务域混合|DWD 事实表 \+ 维度表|汇总表（6\-10 张），按 `stat\_date` 分区|
|**ADS**|业务指标计算、报表数据准备、多表输出|只读 DWS，幂等写入，不跨层依赖|DWS 汇总表|MySQL/ClickHouse 业务表|

### 3\.3 性能规范

#### 3\.3\.1 Shuffle 优化

|场景|推荐做法|禁止做法|
|---|---|---|
|小表关联|Broadcast Join|普通 Join|
|大表聚合|先过滤再聚合|全量聚合后过滤|
|多表关联|先聚合再 Join|先 Join 再聚合|
|分区写入|动态分区 \+ Overwrite|全局排序后写入|

#### 3\.3\.2 内存管理

优先使用 `SmartCacheLite` 等智能缓存组件，按访问频率决定是否缓存，避免滥用 `\.cache\(\)` / `\.persist\(\)`。

#### 3\.3\.3 分区设计

- **ODS 层**：`year, month`

- **DWD 层**：`year, month, taxi\_type`

- **DWS 层**：`stat\_date`（或 `year, month`）

- **ADS 层**：MySQL，按需建立索引

#### 3\.3\.4 数据倾斜处理

- 开启 `spark\.sql\.adaptive\.skewJoin\.enabled = true`

- 对于已知热点键，可在 DWD 层添加随机前缀打散

- 倾斜 Stage 超过 3 倍平均任务时间应告警

### 3\.4 质量规范

#### 3\.4\.1 检查阈值（配置化）

|检查项|推荐阈值|处理方式|
|---|---|---|
|数据留存率|70% \~ 110%|超出告警|
|空值率|\&lt; 5%|超出告警|
|唯一性|≥ 99\.9%|超出输出死信|
|值域异常|\&lt; 1%|超出输出死信|
|重复记录|\&lt; 0\.01%|超出告警|

> 所有阈值必须通过 `ConfigManager` 配置，禁止硬编码。
> 
> 

#### 3\.4\.2 死信处理

**应当写入死信的场景**：

1. 唯一键冲突

2. 值域异常（超出合理范围）

3. 维度关联失败

4. 格式解析错误

**死信表建议字段**：

- `fail\_reason`

- `fail\_time`

- `source\_table`

- `batch\_date`

- `raw\_data`（JSON）

死信表统一存储在 `nyc\_taxi\_dead\_letter` 数据库，按源表名分表：`dead\_letter\_\{source\_table\}`。

#### 3\.4\.3 一致性校验规范

- 跨层数据比对（如 DWS 日总行程数 vs DWD 明细计数）误差应 \&lt; 0\.01%

- 关键指标（总营收、总行程）必须满足 SUM \(明细\) = 汇总值（允许微小舍入误差）

- 一致性校验失败应阻断下游任务（可配置）

## 3\.7 数据质量检测与监控规范（新增）

本节规范如何在离线数仓各层（ODS、DWD、DWS、ADS）集成数据质量检测，并将结果持久化到 MySQL 数据库 `nyc\_taxi\_quality`。

### 3\.7\.1 检测流程总览

```Plain Text
每个 ETL 任务执行完毕后
    │
    ├─ 1. 收集本层质量指标（行数、空值率、唯一性、异常比例等）
    ├─ 2. 写入 data_quality_daily 表（每次执行一条记录）
    ├─ 3. 根据 quality_alert_config 判断是否触发告警
    ├─ 4. 若触发，写入 quality_alert_history 并发送通知
    └─ 5. 可配置是否因质量失败而阻断任务
```

### 3\.7\.2 数据库表结构与字段映射

#### 表 `data\_quality\_daily`

|字段|说明|取值示例|
|---|---|---|
|`check\_date`|数据日期（业务日期）|`2025\-03\-15`|
|`table\_name`|检测的表名（完整）|`nyc\_taxi\_dwd\.fact\_taxi\_trips`|
|`check\_type`|检测类型|`row\_count` / `null\_rate` / `uniqueness` / `range\_anomaly` / `retention\_rate`|
|`check\_status`|通过状态|`PASS` / `WARN` / `FAIL`|
|`expected\_value`|期望值（阈值）|`1000000`（行数期望）|
|`actual\_value`|实际值|`998500`|
|`deviation\_rate`|偏差率（可选）|`\-0\.0015`|
|`detail\_json`|附加详细信息（JSON）|`\{\&\#34;null\_fields\&\#34;: \{\&\#34;vendor\_id\&\#34;: 120, \&\#34;fare\_amount\&\#34;: 35\}\}`|

**检测类型定义**：

|`check\_type`|含义|`expected\_value` 含义|`actual\_value` 含义|
|---|---|---|---|
|`row\_count`|最终行数|最小预期行数|实际行数|
|`retention\_rate`|留存率（与上一层对比）|期望留存率（如 0\.85）|实际留存率|
|`null\_rate`|某字段的空值率|最高允许空值率（如 0\.05）|实际空值率|
|`uniqueness`|唯一键唯一性比例|最低允许唯一比例（如 0\.999）|实际唯一比例|
|`range\_anomaly`|值域异常比例|最高允许异常比例（如 0\.01）|实际异常比例|
|`consistency`|跨层一致性（如 DWS vs DWD）|允许最大误差（如 0\.0001）|实际相对误差|

#### 表 `quality\_alert\_config`

用于配置告警规则，**无需代码修改，支持动态调整**。

|字段|说明|
|---|---|
|`check\_type`|关联的检测类型|
|`table\_name`|可指定具体表，NULL 表示所有表|
|`threshold\_type`|`ABSOLUTE`（绝对值）或 `RATIO`（比率）|
|`warning\_threshold`|触发 WARNING 的阈值|
|`critical\_threshold`|触发 CRITICAL 的阈值|
|`webhook\_url` / `email\_recipients`|通知渠道|

#### 表 `quality\_alert\_history`

自动记录每次触发的告警，`is\_resolved` 由运维人员手动标记。

#### 表 `realtime\_fee\_composition` / `realtime\_hotspot\_topn`

这两张表是**实时模块**的输出，离线模块不直接写入。离线质量检测规范暂不涉及。

### 3\.7\.3 各层检测时机与指标

#### ODS 层（数据加载完成后）

|检测项|目的|失败处理|
|---|---|---|
|`row\_count`|确保源数据非空|阻断任务（空数据会覆盖已有分区）|
|`null\_rate`（关键字段：VendorID, pickup\_datetime, PULocationID）|检测源数据完整性|输出死信 \+ 告警|
|`range\_anomaly`（行程距离、金额基础范围）|过滤明显脏数据（但不删除，仅记录）|仅记录，不阻断|

#### DWD 层（清洗完成后）

|检测项|目的|失败处理|
|---|---|---|
|`retention\_rate`（对比 ODS 行数）|监控清洗损失率|超出阈值告警|
|`null\_rate`（最终表的核心字段）|确保清洗后无残留空值|告警 \+ 输出死信|
|`uniqueness`（`trip\_id`）|确保主键唯一|告警 \+ 去重落盘|
|`range\_anomaly`（金额、距离、乘客数、时长）|值域合规|输出死信 \+ 告警|

#### DWS 层（汇总完成后）

|检测项|目的|失败处理|
|---|---|---|
|`row\_count`|每个汇总表行数合理性（预期行数 = 天数 × 维度基数）|告警|
|`consistency`（DWS 总行数 / 总收入 vs DWD 明细）|确保汇总准确|阻断任务（严重不一致）|
|`null\_rate`|维度字段不应有空（如 borough, zone\_name）|告警 \+ 填充默认值|

#### ADS 层（写入 MySQL 前）

|检测项|目的|失败处理|
|---|---|---|
|`row\_count`|预期最小行数（天数 × 业务粒度）|告警，但允许继续写入|
|`uniqueness`（唯一键）|确保无重复（允许 0\.1% 误差）|告警 \+ 去重后写入|
|`null\_rate`（数值字段应填充 0，字符串填充 “未知”）|符合 ADS 层规范|自动填充，不告警|

### 3\.7\.4 代码集成规范

#### 统一检测接口

```scala
// 定义在 com.taxi.etl.quality.QualityReporter
object QualityReporter {
  def recordQualityCheck(
    tableName: String,
    checkDate: String,
    checkType: String,
    expectedValue: Double,
    actualValue: Double,
    status: String,  // "PASS", "WARN", "FAIL"
    detailJson: Option[String] = None
  ): Unit = {
    // 写入 MySQL data_quality_daily
    // 同时根据 quality_alert_config 触发告警
  }
}
```

#### 各层调用示例

```scala
// DWD 层完成后
val finalCount = enrichedDf.count()
val originalCount = ... // ODS 行数
val retention = finalCount.toDouble / originalCount

QualityReporter.recordQualityCheck(
  tableName = "nyc_taxi_dwd.fact_taxi_trips",
  checkDate = startDate,
  checkType = "retention_rate",
  expectedValue = ConfigManager.getDouble("quality.retention.min", 0.70),
  actualValue = retention,
  status = if (retention >= 0.70) "PASS" else "FAIL",
  detailJson = Some(s"""{"original":$originalCount,"final":$finalCount}""")
)
```

#### 集成到现有代码

- 在 `DwdLayerBuilder`、`DwsLayerBuilder`、`AdsLayerBuilder` 的 try 块末尾，调用 `QualityReporter` 记录各项指标。

- 在 `QualityManager\.fullCheck` 中，除了原有日志和死信，额外调用 `recordQualityCheck`。

- 在 `ConsistencyValidator` 中，将一致性比对结果也写入 `data\_quality\_daily`（`check\_type = \&\#34;consistency\&\#34;`）。

### 3\.7\.6 与现有代码的兼容性

### 3.9 修复日志（2026-05-12）

#### 修复记录 1：Green表分区格式问题

**问题描述：**
日志显示 `Input path does not exist: .../taxi_trip_green_ods/year=2025/month=03`，导致 ODS Validator 和 DWD Layer Builder 执行失败。Green表的分区格式为 `month=03`（带前导零），但代码中使用 `col("month").isin(targetMonths: _*)` where `targetMonths = Seq(1, 2, 3)` 无法匹配 `03`。

**受影响的文件：**
- `src/main/scala/com/taxi/etl/dwd/DwdLayerBuilder.scala`
- `src/main/scala/com/taxi/etl/ods/OdsValidator.scala`

**修复方案：**
1. 在 DwdLayerBuilder 中，为 Green表查询添加月份格式化处理：
   ```scala
   val targetMonthStrings = targetMonths.map(m => f"$m%02d")
   val greenDf = spark.table("nyc_taxi_ods.taxi_trip_green_ods")
     .filter(col("year") === targetYear)
     .filter(col("month").cast("string").isin(targetMonthStrings: _*))
   ```

2. 在 OdsValidator 中，分别构造黄表和绿表的月份列表：
   ```scala
   val monthList = months.mkString(",")
   val greenMonthList = months.map(m => f"$m%02d").mkString(",")
   ```

**修复状态：** ✅ 已完成
