# AI智能助手模块修复日志 - 2026年05月15日

## 修复记录

### 2026-05-15 - AI智能助手会话持久化修复

#### 问题描述
AI智能助手模块查询成功后，数据仅存储在内存HashMap中，未持久化到MySQL数据库nyc_taxi_ai库。刷新页面后历史记录丢失。

#### 修复内容

**后端修复 (taxi-analytics-backend)**

1. **修复AiMapper.java列名不匹配问题**
   - `ai_chat_session`表：`created_at`/`updated_at` → `create_time`/`update_time`
   - `ai_message`表：`type` → `sql_text`/`chart_data`，`created_at` → `create_time`，`execution_time` → `execution_time_ms`
   - `ai_favorite_query`表：`favorite_id` → `id`，`created_at` → `create_time`，新增`query_name`/`execution_count`/`last_execute_time`/`update_time`
   - `ai_scheduled_query`表：`task_id` → `id`，`schedule_time` → `schedule_cron`，`status` → `enabled`，新增`push_type`/`push_target`/`last_run_time`/`next_run_time`

2. **更新AiSessionService接口方法签名**
   - `saveMessage`新增`sqlText`、`chartData`参数
   - `saveFeedback`新增`feedbackScore`、`feedbackComment`参数
   - `addFavorite`新增`queryName`参数
   - `removeFavorite`参数从`String favoriteId`改为`Long favoriteId`
   - `createScheduledTask`新增`taskName`、`scheduleCron`、`pushType`、`pushTarget`参数
   - `deleteScheduledTask`参数从`String taskId`改为`Long taskId`

3. **更新AiSessionServiceImpl实现**
   - 实现与数据库表结构匹配的SQL操作
   - `saveMessage`方法同步更新会话的`message_count`和`last_message_time`
   - `createSession`方法自动创建会话并初始化相关字段

4. **更新AiServiceImpl持久化逻辑**
   - 注入`AiSessionService`依赖
   - 删除`sessionStore` HashMap内存存储
   - 重写`storeSession`方法：创建会话（如不存在）并保存用户消息和助手消息到数据库
   - 重写`getSessionHistory`方法：从数据库读取消息历史并转换为`ChatMessage`列表
   - `processQuery`方法在finally块调用持久化的`storeSession`

#### 修复原理

| 问题 | 原因 | 修复方案 |
|------|------|----------|
| 查询记录不保存 | `storeSession`方法使用HashMap内存存储 | 改为调用`AiSessionService.saveMessage`持久化到MySQL |
| 刷新记录丢失 | 内存存储在应用重启后清空 | 数据持久化到nyc_taxi_ai库 |
| 列名不匹配 | Mapper SQL使用旧列名 | 更新为yc_taxi_ai.csv定义的列名 |
| 会话不自动创建 | 新会话不会自动插入数据库 | 在`storeSession`中检查并创建会话 |

✅ 编译通过：
- 后端Maven编译成功：`mvn clean compile -DskipTests` BUILD SUCCESS
- 前端Vite构建成功：`npm run build` built in 26.58s

#### 影响范围

- 不影响其他功能模块（AiDbService、AiDbMapper等保持独立）

### 2026-07-19 - 实时监控模块数据展示修复

#### 问题描述
实时监控模块存在以下数据展示问题：
1. **热点区域显示ID而非名称**：热点区域列表显示的是区域ID（如74、75），用户无法识别具体区域名称
2. **支付方式分布数据异常**：饼图显示多个"信用卡: 0"标签，数据值为0，无法正常展示支付方式分布
3. **实时趋势数据范围问题**：趋势图未限制时间范围，可能返回过旧的数据

#### 修复内容

**后端修复 (taxi-analytics-backend)**

1. **新增ZoneMapping配置类 (config/ZoneMapping.java)**
   - 创建NYC出租车区域ID到名称的映射表（包含312个区域）
   - 提供 `getZoneName(String zoneId)` 方法进行ID到名称的转换
   - 支持常见区域如Midtown Center、Midtown East、Upper East Side等

2. **更新RealtimeServiceImpl.java**
   - 在热点区域数据返回前，添加区域名称映射
   - 为每个热点记录添加 `zoneName` 字段

3. **修复RealtimeMapper.java**
   - `getFeeComposition()` 查询添加 `GROUP BY payment_type` 和 `COUNT(*) as trip_count`
   - `getTrend()` 查询添加 `WHERE window_end >= DATE_SUB(NOW(), INTERVAL 24 HOUR)` 时间范围限制

**前端修复 (taxi-analytics-frontend)**

1. **修复Index.vue支付方式图表**
   - 过滤数据值为0或无效的记录
   - 修正数据绑定逻辑，优先使用 `trip_count` 字段，回退到 `total_fare`

#### 修复原理

| 问题 | 原因 | 修复方案 |
|------|------|----------|
| 热点区域显示ID | 数据库存储的是区域ID数字，未映射为名称 | 创建ZoneMapping配置类，在Service层添加名称映射 |
| 支付方式分布异常 | SQL查询缺少GROUP BY，返回多条相同payment_type记录且无count字段 | 添加GROUP BY和COUNT聚合 |
| 趋势数据范围过大 | 未限制时间范围，返回所有历史数据 | 添加24小时时间范围过滤 |
